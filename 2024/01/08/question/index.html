Redis持久化：

为什么需要持久化？

Redis 是内存数据库，读写速度极快，但内存是易失性存储。如果只依赖内存，一旦进程退出或机器重启，所有数据都会消失。持久化就是为了解决数据安全与数据恢复的问题。

方式一、RDB全称Redis DataBase Backup file（Redis数据备份文件），也被叫做Redis数据快照。简单来说就是把内存中的所有数据都记录到磁盘中。当Redis实例故障重启后，从磁盘读取快照文件，恢复数据。

Redis的数据过期策略：

如何发现并删除已经过期的 Key？

方式一、惰性删除：设置该key过期时间后，我们不去管它，当需要该key时，我们在检查其是否过期，如果过期，我们就删掉它，反之则返回该key。

优点：对CPU友好，只会子啊使用该key时才会进行过期检查，对于很多用不到的key不用浪费时间进行检查。

缺点：对内存不友好，如果一个key已经过期，但是一直没有使用，那么该key就会一直存放在内存中，内存永远不会释放。

方式二、定期删除：每隔一段时间，我们就对一些key进行检查，删除里面过期的key（从一定数量的数据库中取出一定数量的随机key进行检查，并删除其中过期的key）

定期清理有2种模式：

SLOW模式是定时任务，执行频率默认为10hz，每次不超过25ms，可以通过修改配置文件redis.conf的hz选项来调整次数

FAST模式执行频率不固定，但两次间隔不低于2ms，每次耗时不超过1ms

优点：可以通过限制删除操作执行的时长和频率来减少对CPU的影响。另外定期删除，也能有效释放过期键占用的内存。

缺点：难以确定删除操作执行的时长和频率。

Redis的过期策略：惰性删除 + 定期删除 两种策略进行配合使用！！！。

缓存穿透：

缓存穿透是查询一个不存在的数据，mysql查询不到数据也不会直接写入缓存，就会导致每次请求都查数据库。

解决方案一：缓存空数据，查询返回的数据为空，仍吧这个空结果进行缓存。

缺点：消耗内存、可能会发生数据不一致的问题（刚开始时候数据确实是没有id为100得到数据，但后面有了）

解决方案二：布隆过滤器



误判率是一定会有的，一般是可以通过参数设置，一般设置为5%以下。

缓存击穿：

缓存击穿是给某一个key设置了过期时间，当key过期的时候，恰好这个时间点对这个key有大量的并发请求过来，这些并发的请求可能会瞬间把数据库压垮。

解决方案一：互斥锁

优缺点：数据强一致、性能差

解决方案二：逻辑过期

优缺点：高可用、性能优，但是不保证数据一致

缓存雪崩：

缓存雪崩是指在同一段大量的缓存key同时失效或者Redis服务宕机，导致大量请求到达数据库，带来巨大压力

解决方案：

一、给不同的Key的过期时间设置随机值

二、利用Redis集群提高服务的可用性

三、给缓存业务添加降级限流策略

四、给业务添加多级缓存

事务的传播特性

Spring 中定义的 7 种事务传播行为

传播行为 含义
REQUIRED（默认） 如果当前有事务，就加入它；如果没有，就新建一个。适用于正删改查
SUPPORTS 如果当前有事务，就加入；如果没有，就以非事务方式执行。适用于查询
REQUIRES_NEW 始终新建一个独立的事务；如果当前有事务，将其挂起。适用于日志
——————以下比较少用
MANDATORY 必须在一个已有事务中执行，否则抛出异常。
NOT_SUPPORTED 以非事务方式执行；如果当前有事务，将其挂起。
NEVER 以非事务方式执行；如果当前有事务，抛出异常。
NESTED 如果当前有事务，则在嵌套事务内执行；否则行为类似 REQUIRED。

事务失效的原因：

事务失效的原因：同一个类内部方法调用（无事务）

原因：Spring 事务通过代理对象执行，this.methodB() 绕过了代理。
解决：注入自己或通过 AopContext.currentProxy()。

方法不是 public 的

原因：Spring 默认使用 JDK 动态代理或 CGLIB，只拦截 public 方法。

异常被 catch 吞掉

原因：事务管理器只对未捕获的运行时异常（或特定异常）回滚。

吞掉抛出的异常类型不对

解决：@Transactional(rollbackFor = Exception.class)

没有在 Spring 容器中管理

没有在 Spring 容器中管理


原因：代理机制失效，必须是 Spring Bean。

数据库/表不支持事务

不同线程中的事务隔离

原因：Spring 事务基于 ThreadLocal 存储连接，子线程拿不到。

慢SQL分析

通过Explain分析SQL语句


possibly_key:当前sql可能会用到的索引
key:当前sql实际命中的索引
key_len:索引占用大小
extra:额外的优化建议，Using where;Using index查找使用了索引，需要在数据都在的索引列中能找到，不需要回表查询数据，Using index condition 查找使用了索引，但是需要回表查询数据
type:当前sql的连接类型，性能由好较差为NULL、system、const、eq_ref、ref、range、index、all

索引失效的形式有哪些

隐式类型转换

— 假设 phone 字段是 varchar 类型
EXPLAIN SELECT * FROM user WHERE phone = 13800000000; — ❌ 失效

— 正确写法（字符串匹配）
EXPLAIN SELECT * FROM user WHERE phone = ‘13800000000’; — ✅ 可用索引
原因：MySQL 会将 13800000000 转为字符串再比较，但函数作用于字段，导致无法使用索引。

其他例子：

字段是 int，传入 ‘123’（字符串） 正常 ✅（自动转换的是值不是字段）

在索引列上使用函数或计算

EXPLAIN SELECT FROM user WHERE DATE(create_time) = ‘2026-05-08’; — ❌ 失效
EXPLAIN SELECT FROM user WHERE create_time >= ‘2026-05-08’ AND create_time < ‘2026-05-09’; — ✅ 可用索引

EXPLAIN SELECT FROM user WHERE age + 10 = 30; — ❌ 失效
EXPLAIN SELECT FROM user WHERE age = 20; — ✅ 可用索引

LIKE 以通配符开头

EXPLAIN SELECT FROM user WHERE name LIKE ‘%张三%’; — ❌ 失效
EXPLAIN SELECT FROM user WHERE name LIKE ‘%张三’; — ❌ 失效
EXPLAIN SELECT * FROM user WHERE name LIKE ‘张三%’; — ✅ 可用索引（前缀匹配）

使用 OR 条件

— 假设 name 有索引，age 没有索引
EXPLAIN SELECT * FROM user WHERE name = ‘张三’ OR age = 20; — ❌ 索引失效（可能全表扫描）

— 改写为 UNION
SELECT FROM user WHERE name = ‘张三’
UNION
SELECT FROM user WHERE age = 20; — ✅

复合索引不满足最左前缀原则

— 创建复合索引 (a, b, c)
CREATE INDEX idx_a_b_c ON t(a, b, c);

— ✅ 符合最左前缀，可用索引
WHERE a = 1
WHERE a = 1 AND b = 2
WHERE a = 1 AND b = 2 AND c = 3

— ❌ 不符合最左前缀，索引失效
WHERE b = 2
WHERE c = 3
WHERE b = 2 AND c = 3

使用 !=、<>、NOT IN、NOT EXISTS

EXPLAIN SELECT FROM user WHERE age != 20; — ❌ 大概率失效
EXPLAIN SELECT FROM user WHERE age <> 20; — ❌ 失效
EXPLAIN SELECT * FROM user WHERE age NOT IN (20, 30); — ❌ 失效

Eureka和Nacos的区别

Eureka 只做服务注册与发现，而 Nacos 在此基础上还集成了强大的配置中心功能

Eureka 只支持 AP（高可用）模式，牺牲了数据强一致性，在极端情况下可能读到已死亡的实例。Nacos 则支持 AP 和 CP 两种模式动态切换，可以兼顾分布式环境下的可用性和数据一致性。

Eureka 默认拉取间隔是30秒，服务摘除需要等到90秒心跳超时，客户端感知服务变更的平均时延在秒级甚至分钟级。Nacos 采用了推拉结合的模式，默认能在10秒内完成服务变化的推送，响应更快。

SpringCloud中的中间件：

服务注册与发现：Eureka、Nacos

服务之间调用：RestTemplate + Ribbon、OpenFeign

网关：Zuul、Spring Cloud Gateway

消息中间件：RabbitMQ、RocketMQ、Kafka

Rabbit-MQ是如何保证消息不丢失？

消息丢失可能发生在：
1、生产者->交换机时可能丢失
2、交换机->队列时可能丢失
3、队列->消费者时候可能丢失
回答：
1、开启生产者确认机制，确保生产者的消息能到达队列，如果报错可以先记录到日志中，再去修复数据。
2、开启持久化功能，确保消息未消费前在队列中不会丢失，其中交换机、队列、和消息都要做持久化
3、开启消费者确认机制为auto，由Spring确认消息处理成功后完成ack，当然也需要设置一定的重试次数
，我们当时设置了3次还没有收到消息，就将失败后的消息投递到异常交换机，交由人工处理。

RabbitMQ提供了publisher confirm机制来避免消息发送到MQ过程中丢失。消息发送到MQ以后，会返回一个结果给发送者，表示消息是否处理成功。


如图可以看到：
当消息发送成功时候:生产者会接到消息：ack publish-confirm
当交换机接受消息失败时候：nack publish-confirm
当队列接受消息失败时候：ack publish-return

消息失败之后如何处理呢？
1、回调方法即时重发
2、记录日志
3、保存到数据库然后定时重发，成功发送后即刻删除表中的数据

MQ宕机时候呢？

消息持久化：MQ默认是内存消息，开启持久化功能可以确保缓存中在MQ中的消息不丢失。
